# 事例

S社の主要事業は鉄製品の製造販売。
鉄生産統合システムを所持し、改修している。
サブシステムの売上収支管理機能の改善が目下の課題。

BA社はS社のシステム改修に長年参加している。
案件内容
詳細設計～結合試験
期間は5ヶ月
※懸念点として、精査したところ5ヶ月では足りない。

S社側の確認や検証、既存設計書の確認、最新化などの作業をWBSで洗い出して説明を実施。7ヶ月で最終合意。
延長2ヶ月分は、S社憲章の並行作業化と次工程開始予備から捻出。

既存体制は、S社にブリッジSEとして参加させていたB氏に小口案件を頼むことでワンストップによる開発が進む仕組みを構築していた。
今回は、規模が大きいため、S社のH氏の配下にB氏をつけ、S社の立場として支援する形の体制とした。

- 契約形態：一括請負
- 契約金額：4300万円
- マイルストーン
  - 詳細設計：1.5m
  - 製造：2.0m
  - 単体試験：2.0m
  - 結合試験：1.5m

システム構成

- ASP.NET(MVC) SQL Server HTML5+CSS3, IIS, SVF
規模
- 51.7KL

QCDSのうち、Dを最優先とする。
発生課題

- 既存設計書の品質問題  
  既存機能の改修について、既存詳細設計書に不備あり  
  ソースとの不一致、つぎはぎ、資料粒度のバラツキ
  →ソースから詳細設計を最新化。工数は見積もり範囲内に収まった。
- 詳細設計の仕様変更
  納期優先のため仕様を最小化。
  →予算内での現実案を提案、それでも必要な場合は切り出して準委任契約化

## 成功要因

- QCDSの優先項目を早期形成できたこと。
- 工数付属をWBSを用いて説得できたこと。
  - 7ヶ月の根拠を持って定量的な営業ができたこと。
- 仕様変更の現実案を提示して能動的な変更に持っていけたこと。
  - さらに、足が出る部分は準委任契約にできたこと

## 改善箇所

### 設計書の品質問題

既存設計書の品質問題について、構造的課題がそのままであること。  
これにより再発の可能性が高い。  
BA社側が常に肩代わりし続ける構造になりかねない。

### 初期見積もりの甘さ

工期見積もりの甘さ  
WBSの精査で2ヶ月の延長を行ったが、そもそもリスク分析が甘い可能性あり。  
見積もり工程における情報不足や見積もりプロセスに課題がある可能性あり。  

WBSで巻き返しただけで、同じ問題が発生するのはさけるべき。

### 一括請負における仕様変更

一括請負は、仕様凍結が前提。  
詳細設計から請け負う前に、要件定義と仕様締結の契約を行ってFIXさせるべき。  
変更のたびにコストを負担させられる可能性がある。

### S社側の体制依存が高く、個人に寄り過ぎ

B氏の属人性依存が高い。

### 技術スタックがレガシー寄り

使ってる技術が古い。  
BP、若手エンジニアの参画難などが発生する。
